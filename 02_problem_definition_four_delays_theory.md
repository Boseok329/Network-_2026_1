# 네트워크 지연의 4가지 원인 (이론 분석)

> 📌 **이 문서의 목적**: Module 1에서 배운 4대 지연 요소를 HFT 관점에서 깊이 있게 분석한다. 각 지연의 물리적 의미, 공식의 단위, HFT에서의 비중을 정리한다.
>
> 📚 **출처**: Kurose & Ross, *Computer Networking: A Top-Down Approach*, 8th ed., Chapter 1.4
> 📚 **강의자료**: `CN_Module1_Session1_overview.pdf`

## 0. 전체 지연 공식

```
d_total = d_proc + d_queue + d_trans + d_prop
```

| 기호 | 풀네임 | 한글 | 단위 |
|------|--------|------|------|
| d_total | Total delay | 전체 지연 | seconds |
| d_proc | Processing delay | 처리 지연 | seconds |
| d_queue | Queueing delay | 큐잉 지연 | seconds |
| d_trans | Transmission delay | 전송 지연 | seconds |
| d_prop | Propagation delay | 전파 지연 | seconds |

각 라우터(중간 노드)마다 위 4가지 지연이 발생한다. 따라서 **end-to-end 지연**은 경로 상의 모든 라우터에서 발생하는 4대 지연의 누적이다.

```
d_e2e = Σ (d_proc_i + d_queue_i + d_trans_i + d_prop_i)
```

---

## 1. Processing Delay (d_proc) — 처리 지연

### 1.1 정의

라우터(또는 호스트의 네트워크 스택) 가 패킷을 받아 다음 처리 단계로 넘기기 전에 수행하는 작업에 걸리는 시간:

1. **비트 에러 체크** (체크섬 검증)
2. **출력 링크 결정** (포워딩 테이블 참조)
3. **헤더 분해 및 재조립**

### 1.2 일반 환경의 크기

- 일반적으로 **< 1µs** (마이크로초 미만)
- Kurose & Ross에서는 "typically negligible" 로 기술

### 1.3 HFT 환경에서의 의미

- 일반 환경에선 무시되지만, HFT에선 **µs 단위가 누적되면 결정적**
- 호스트 측 d_proc은 대부분 **OS 커널 오버헤드**:
  - 인터럽트 처리: ~5µs
  - 컨텍스트 스위치: ~3µs
  - TCP/IP 스택 통과: ~10µs
  - 사용자 공간 복사: ~2µs

### 1.4 줄이는 방법

| 기법 | 효과 |
|------|------|
| **Kernel Bypass (DPDK)** | 인터럽트, 커널 스택 우회 → ~1µs로 단축 |
| **Busy-polling** | 인터럽트 대신 NIC를 계속 감시 → 지터 제거 |
| **CPU Pinning** | 컨텍스트 스위치 제거 |
| **FPGA NIC** | TCP/UDP 처리 자체를 하드웨어에 → 100ns 미만 |
| **RDMA** | CPU 개입 없이 메모리 직접 접근 |

> 🔬 김정하 (Protocol Lead) 의 분석 영역

---

## 2. Queueing Delay (d_queue) — 큐잉 지연

### 2.1 정의

패킷이 라우터의 출력 큐(버퍼) 에서 자기 차례를 기다리는 시간.

```
d_queue = (큐 안에 있는 비트 수) / (출력 링크 속도)
```

### 2.2 가변성

- **가장 변동성이 큰 지연 요소** — 트래픽 부하에 따라 0부터 무한대까지 변동
- 평균 대기 시간은 **트래픽 강도 (La/R)** 에 의존
  - 자세한 분석은 [`traffic_intensity_analysis.md`](./traffic_intensity_analysis.md) 참조

### 2.3 HFT 환경에서의 의미

- HFT의 **진짜 적**:
  - 평소엔 작지만 변동성 시즌(FOMC, 실적 발표) 에 폭발
  - 정작 가장 돈이 되는 순간에 가장 느려짐
- 큐가 가득 차면 **패킷 드롭(loss) 발생** → 프로토콜 복구 메커니즘 발동
  - **TCP**: RTO 타이머 후 재전송 → 지연 폭등
  - **UDP**: 포기, 다음 신호로 진행

### 2.4 줄이는 방법

| 기법 | 효과 |
|------|------|
| **QoS / Priority Queuing** | HFT 트래픽에 우선순위 부여 |
| **버퍼 크기 축소** | 지연은 줄지만 손실 가능성 증가 (Bufferbloat 회피) |
| **Direct Cross-Connect** | 라우터 자체를 우회 (Co-lo) |
| **RDMA Verbs** | 큐 시스템 자체를 우회 |

---

## 3. Transmission Delay (d_trans) — 전송 지연

### 3.1 정의

패킷의 **모든 비트** 를 링크 위에 밀어 올리는 데 걸리는 시간.

```
d_trans = L / R
```

- L: 패킷 길이 (bits)
- R: 링크 대역폭 (bps)

### 3.2 비유 (강의에서 배운)

> 톨게이트를 차들이 빠져나가는 데 걸리는 시간

- d_trans = 차(패킷) 가 톨게이트(링크 입구) 를 통과하는 시간
- d_prop = 차가 고속도로(링크) 를 달리는 시간
- 두 개념은 **완전히 다르다**

### 3.3 단위와 예시

| 패킷 크기 (L) | 링크 (R) | d_trans |
|--------------|---------|---------|
| 64 bytes (HFT 거래 메시지) | 1 Gbps | 0.512 µs |
| 64 bytes | 10 Gbps | 0.0512 µs |
| 64 bytes | 100 Gbps | 0.00512 µs |
| 1500 bytes (일반 MTU) | 1 Gbps | 12 µs |

### 3.4 HFT 환경에서의 의미

- HFT 메시지는 **매우 작음** (주문/체결 신호 ~수십 byte)
- 10/40/100 GbE NIC 이미 보편화 → d_trans는 **거의 무시 가능**
- 본 프로젝트에서 큰 변수 아님

### 3.5 줄이는 방법

- 링크 대역폭 R 증가 (10 → 100 GbE)
- 패킷 크기 L 축소 (헤더 압축, 응용 프로토콜 최적화)

## 4. Propagation Delay (d_prop) — 전파 지연

### 4.1 정의

신호(비트) 가 물리적 매체를 따라 이동하는 시간.

```
d_prop = d / s
```

- d: 물리적 링크 길이 (m)
- s: 매체에서의 전파 속도 (m/s)

### 4.2 매체별 전파 속도

| 매체 | 전파 속도 | 광속 대비 |
|------|---------|---------|
| 진공 | 3.0 × 10⁸ m/s | 100% |
| 공기 (마이크로웨이브) | ~3.0 × 10⁸ m/s | ~100% |
| 광케이블 (싱글모드 광섬유) | ~2.0 × 10⁸ m/s | 67% |
| 구리선 (전기 신호) | ~2.0 × 10⁸ m/s | 67% |

### 4.3 거리별 d_prop 예시 (광케이블 기준)

| 거리 (d) | d_prop |
|---------|--------|
| 1 m (Co-location 내부) | 5 ns |
| 1 km (도시 내) | 5 µs |
| 1,300 km (시카고-뉴욕 직선) | 6.5 ms |
| 5,000 km (뉴욕-런던 직선) | 25 ms |

### 4.4 HFT 환경에서의 의미

- **물리 법칙의 한계** — 빛의 속도가 절대 상한선
- 코드로 절대 줄일 수 없는 영역
- 전체 지연 중 **장거리에서 가장 큰 비중**:
  - 시카고 ↔ 뉴욕: 전체 RTT의 약 99%가 d_prop
- 이 때문에 HFT 인프라 투자의 대부분이 d_prop 단축에 집중됨

### 4.5 줄이는 방법

| 기법 | 효과 |
|------|------|
| **Co-location** | 거리 d 자체를 수 m로 축소 (가장 효과적) |
| **직선 광케이블** | 거리 d 단축 (Spread Networks) |
| **마이크로웨이브** | 매체의 전파 속도 s 향상 (광속의 67% → 99%) |
| **위성 통신 (LEO)** | 대륙 간 우회 단축 (실험 단계) |

## 5. 4대 지연의 HFT 환경 비중 비교

### 5.1 시나리오별 비중

#### Case A: Co-location 환경 (거리 ~1m)

| 지연 | 크기 | 비중 |
|------|------|------|
| d_prop | 5 ns | 미미 |
| d_trans (64B @ 10G) | 51 ns | 작음 |
| d_proc (커널 우회) | ~1 µs | 큼 |
| d_queue (저부하) | ~1 µs | 큼 |
| **합계** | **~2 µs** | — |

→ 호스트 내부 처리(d_proc, d_queue) 가 지배적

#### Case B: 시카고 ↔ 뉴욕 (거리 1,300km)

| 지연 | 크기 | 비중 |
|------|------|------|
| d_prop (광케이블) | 6.5 ms | **99%** |
| d_trans | < 1 µs | 미미 |
| d_proc | ~수 µs | 미미 |
| d_queue | ~수 µs | 미미 |
| **합계** | **~6.5 ms** | — |

→ 전파 지연이 압도적

### 5.2 결론: 거리에 따라 우선순위가 다르다

- **단거리 (Co-lo, 데이터센터 내부)**: d_proc, d_queue가 핵심 → 우리 프로젝트의 영역
- **장거리**: d_prop이 압도적 → 인프라 투자 영역

## 6. 본 프로젝트의 분석 범위

| 지연 | 통제 가능성 | 본 프로젝트 분석 |
|------|----------|--------------|
| d_prop | ❌ (물리 한계) | 분석 안 함 (설명만) |
| d_proc | △ (커널 바이패스) | 김정하 (Protocol Lead) |
| d_trans | ✅ (이미 해결) | 분석 안 함 |
| **d_queue + 손실 복구** | ✅ **핵심 통제 지점** | **본 프로젝트의 메인 타겟** |
